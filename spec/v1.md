# AgentWharf Hub Protocol v1

## Status

This document is the authoritative v1 protocol specification for AgentWharf.
It defines the WebSocket protocol used by clients, adapters, and the Hub to
synchronize long-running agent sessions.

The protocol is provider-neutral and host-neutral. AgentWharf does not define
how a host is created, metered, or administered. Deployments can map their own
host concepts to AgentWharf sessions outside this protocol.

## Goals

1. Provide a durable per-session event stream with Hub-issued sequence numbers.
2. Support multiple clients observing and controlling the same session.
3. Allow clients and adapters to reconnect without losing durable events.
4. Keep provider-specific behavior out of the Hub by using adapters.
5. Keep self-hosting simple: one Hub process, one adapter process, and a local
   EventStore are enough for a complete setup.

## Non-Goals

1. Defining the provider-side agent protocol. Adapters should use ACP when a
   provider supports it and fallback transports only when needed.
2. Moving large file contents through WebSocket frames. Large blobs should use a
   separate content-addressed endpoint.
3. Multi-Hub coordination. v1 assumes one authoritative Hub per session.
4. End-to-end encryption. v1 relies on TLS, scoped tokens, and a trusted Hub.

## Roles

| Role | Responsibility |
| --- | --- |
| Hub | Authenticates connections, assigns durable event sequence numbers, persists durable events, fans out events, routes commands, and replays gaps. |
| Adapter | Bridges a provider process to the Hub protocol and reports provider activity as normalized session events. |
| Client | Subscribes to event streams and sends commands such as user messages, approval decisions, interrupts, and stop requests. |

Rules:

- The Hub never executes provider commands. Execution is adapter-side only.
- The adapter never talks directly to clients. All traffic goes through the Hub.
- Durable event `seq` values are assigned only by the Hub after persistence.

## Transport

Transport is WebSocket over TLS (`wss://`) with JSON text frames.

One connection has one role:

- A client connection can subscribe to one or more sessions.
- An adapter connection serves exactly one session.

Future binary encodings can be negotiated in a later protocol version.

## Frames

| Frame | Direction | Purpose |
| --- | --- | --- |
| `hello` | Peer -> Hub | First frame. Authenticates, declares role, negotiates protocol version, and optionally supplies replay cursors. |
| `hello.ack` | Hub -> peer | Accepts the connection and returns session summaries. |
| `event` | Adapter -> Hub, Hub -> clients | Session event. Durable events have `seq`; ephemeral events omit `seq`. |
| `command` | Client -> Hub -> adapter | Idempotent operation identified by `cmd_id`. |
| `command.ack` | Hop-by-hop | Acknowledges a command as `accepted`, `rejected`, or `duplicate`. |
| `ping` | Either direction | Heartbeat request. |
| `pong` | Either direction | Heartbeat response. |
| `error` | Hub -> peer | Protocol error with a machine-readable code. Fatal errors are followed by connection close. |

## Handshake

Client `hello`:

```json
{
  "frame": "hello",
  "protocol_version": 1,
  "role": "client",
  "token": "<scope-bound token>",
  "subscriptions": [
    { "session_id": "ses_01H8X", "last_seq": 41 }
  ]
}
```

Adapter `hello`:

```json
{
  "frame": "hello",
  "protocol_version": 1,
  "role": "adapter",
  "token": "<session-bound adapter token>",
  "session_id": "ses_01H8X",
  "provider": "claude-code",
  "resume": true
}
```

Client `hello.ack`:

```json
{
  "frame": "hello.ack",
  "protocol_version": 1,
  "sessions": [
    {
      "session_id": "ses_01H8X",
      "state": "ready",
      "provider": "claude-code",
      "latest_seq": 57,
      "replay_from": 42
    }
  ]
}
```

Version negotiation:

- Peers send the highest protocol version they support.
- The Hub selects the highest mutually supported version.
- If no mutually supported version exists, the Hub sends `error` and closes.

## Persistence And Replay Rules

These rules are normative:

```text
R1  seq is a per-session monotonically increasing integer starting at 1.
    The Hub assigns seq during durable event persistence.
R2  Durable events must be successfully appended to the EventStore before
    they are broadcast. If persistence fails, the event does not exist.
R3  Ephemeral events have no seq, are not persisted, are not replayed, and
    may be dropped without breaking protocol correctness.
R4  EventStore implementations must enforce UNIQUE(session_id, seq).
R5  Replay returns all durable events in (last_seq, latest_seq] ordered by
    seq, then continues with the live stream without gaps or duplicates.
```

Durable event envelope:

```json
{
  "frame": "event",
  "type": "session.message",
  "session_id": "ses_01H8X",
  "seq": 42,
  "time": 1764937200123,
  "payload": {}
}
```

Ephemeral event envelope:

```json
{
  "frame": "event",
  "type": "log.tail",
  "session_id": "ses_01H8X",
  "time": 1764937200123,
  "payload": {}
}
```

`time` is Unix time in milliseconds.

## Durable Events

| Type | Producer | Meaning |
| --- | --- | --- |
| `session.state` | Adapter or Hub | Session state transition. |
| `session.message` | Adapter or Hub | Conversation message. Hub records user messages before routing `session.send`. |
| `session.tool_call` | Adapter | Tool call start or result. |
| `permission.request` | Adapter | Provider asks for approval before an action. |
| `permission.decision` | Hub | User or policy decision for a permission request. |
| `file.changed` | Adapter | Workspace file change summary. Optional in the v1 minimum implementation. |
| `session.error` | Adapter or Hub | Non-recoverable error detail. |
| `x.*` | Any peer allowed by policy | Extension namespace. The Hub persists and forwards durable extension events but does not interpret them. |

## Ephemeral Events

| Type | Producer | Meaning |
| --- | --- | --- |
| `agent.activity` | Adapter or Hub | Real-time activity summary. Not replayed. |
| `log.tail` | Adapter or Hub | Terminal or build log tail. Not replayed. |
| `session.idle_warning` | Hub | Session-scoped warning that the backing VM will soon be suspended for idleness. Not persisted or replayed. |

### `session.idle_warning`

```json
{
  "vm_id": "vm_01H8X",
  "seconds_until_suspend": 600,
  "suspend_at": "2026-06-21T12:30:00Z",
  "message": "This VM will pause soon because it has been idle."
}
```

Fields:

- `seconds_until_suspend`: non-negative seconds from Hub send time to expected suspend time.
- `suspend_at`: RFC3339 UTC timestamp.
- The event is ephemeral: it has no `seq`, MUST NOT be written to EventStore,
  and is not replayed after reconnect.
- Durable platform notification delivery uses `vm.idle_10m`; it does not change
  Hub event sequencing.

### `session.message`

```json
{
  "message_id": "msg_01H8Y",
  "role": "user",
  "content": [
    { "kind": "text", "text": "Continue" },
    {
      "kind": "attachment",
      "blob": "sha256:ab12",
      "name": "mock.png",
      "media_type": "image/png"
    }
  ],
  "in_reply_to": "msg_01H8X"
}
```

Fields:

- `role`: `user`, `agent`, or `system`.
- `content`: ordered message parts.
- `in_reply_to`: optional message id.

### `session.tool_call`

```json
{
  "tool_call_id": "tc_01H8Z",
  "phase": "start",
  "name": "bash",
  "input": { "command": "npm install" },
  "result": null
}
```

Result example:

```json
{
  "tool_call_id": "tc_01H8Z",
  "phase": "result",
  "name": "bash",
  "input": null,
  "result": {
    "status": "ok",
    "output_preview": "added 1024 packages",
    "truncated": true
  }
}
```

Large outputs must be represented as previews plus optional blob references.
The maximum event payload size is 64 KiB.

ACP mapping rule:

- When an ACP provider exposes an action through `session/request_permission`
  without also sending an explicit `tool_call` or `tool_use` update, the adapter
  MUST first emit a durable synthetic `session.tool_call` with `phase=start`,
  then emit the existing `permission.request`.
- The synthetic `tool_call_id` MUST be stably derived from the permission
  request id. The recommended format is `permission:<request_id>`.
- The synthetic `name` is the ACP `action`, or `permission` when action is
  absent. The synthetic `input` contains the action, summary, risk level, and
  provider options when present.
- The subsequent `permission.request.detail` still carries provider options and
  provider session id for approval routing and audit.

### `permission.request`

```json
{
  "request_id": "pr_01H90",
  "action": "git.push",
  "risk_level": "medium",
  "summary": "Agent requests permission to push to origin/main",
  "detail": {},
  "expires_at": 1764937800000
}
```

Fields:

- `risk_level`: `low`, `medium`, or `high`.
- `expires_at`: Unix time in milliseconds.

### `permission.decision`

```json
{
  "request_id": "pr_01H90",
  "decision": "approve",
  "decided_by": "usr_01H91",
  "note": ""
}
```

Fields:

- `decision`: `approve`, `deny`, or `expired`.
- `decided_by`: user id or a policy actor such as `policy:auto`.

### `file.changed`

```json
{
  "paths": [
    { "path": "src/app.ts", "change": "modified" },
    { "path": "package-lock.json", "change": "modified" }
  ],
  "batch_id": "fc_01H92",
  "summary": { "modified": 2, "added": 0, "deleted": 0 }
}
```

`change` is `added`, `modified`, or `deleted`.

## Ephemeral Events

Ephemeral events omit `seq` and are never persisted.

| Type | Meaning |
| --- | --- |
| `presence` | Online clients and typing indicators. Optional in the v1 minimum implementation. |
| `agent.activity` | Transient provider activity such as thinking or tool execution. |
| `log.tail` | Terminal or build log tail. |
| `resource.sample` | Instantaneous resource sample. |

## Commands

Command envelope:

```json
{
  "frame": "command",
  "cmd_id": "cmd_01H91",
  "type": "session.send",
  "session_id": "ses_01H8X",
  "payload": {
    "content": [{ "kind": "text", "text": "Continue" }]
  }
}
```

| Type | Direction | Effect |
| --- | --- | --- |
| `session.send` | Client -> Adapter | Sends a user message. The Hub first persists a `session.message` with `role=user`, then routes the command. |
| `permission.respond` | Client -> Hub -> Adapter | Records and routes an approval decision. |
| `session.interrupt` | Client -> Adapter | Interrupts the provider's current action. |
| `session.stop` | Client -> Adapter | Ends the session and asks the provider process to exit gracefully. |

Command acknowledgement:

```json
{
  "frame": "command.ack",
  "cmd_id": "cmd_01H91",
  "status": "accepted",
  "reason": ""
}
```

Rules:

- `cmd_id` is generated by the sender and must be unique for the sender's
  command stream.
- The Hub returns `duplicate` for a previously accepted `cmd_id` and must not
  execute it again.
- `accepted` means the receiving hop accepted the command. Provider-visible
  results are reported later as events.
- When the adapter is offline, the Hub can buffer `session.send` and
  `permission.respond` for up to 64 commands or 10 minutes. Other commands are
  rejected with `adapter_offline`.

## Permission Flow

```text
provider attempts an action
  -> adapter emits a synthetic session.tool_call(phase=start) when the ACP
     request has no explicit tool_call/tool_use update
  -> adapter pauses the provider and emits permission.request
  -> Hub persists the request and broadcasts it
  -> a client sends permission.respond
  -> Hub persists permission.decision and broadcasts it
  -> Hub routes the decision to the adapter
  -> adapter resumes or denies the provider action
```

Rules:

- Only the first valid decision for a `request_id` is accepted.
- Later responses for the same `request_id` return `duplicate`.
- Policy decisions are also persisted as `permission.decision`.
- If a request expires, the Hub persists `decision=expired`; adapters must
  treat expiration as denial.

## Reconnect And Replay

Client reconnect:

```text
client hello includes last_seq
  -> Hub replays durable events in (last_seq, latest_seq]
  -> Hub switches the connection to the live stream
```

Adapter reconnect:

```text
adapter hello includes resume=true
  -> Hub verifies the session binding
  -> Hub accepts the adapter
  -> Hub delivers buffered commands in order
```

If the adapter process restarts with the provider process, recovery is
provider-specific. The adapter should emit `session.state` transitions such as
`recovering -> ready` or `ended` with a crash reason.

## Adapter Session State

Legal state transitions:

```text
starting -> ready
ready -> busy
busy -> ready
busy -> waiting_permission
waiting_permission -> busy
waiting_permission -> ready
ready -> recovering
busy -> recovering
recovering -> ready
any -> ended
any -> error
```

Common `ended.reason` values:

- `user_stop`
- `provider_exit`
- `crash`
- `timeout`

Adapters must:

- Prefer ACP for provider integration.
- Keep provider-specific parsing in provider-specific adapter packages.
- Supervise provider process lifetime with bounded restart attempts.
- Send `ping` every 20 seconds and treat 60 seconds without heartbeat as
  offline.
- Apply secret masking before sending user-visible text to the Hub.

## Scope And Authorization

Tokens are issued outside the Hub. The Hub authenticates tokens and authorizes
requested scopes.

| Scope | Capability | Typical holder |
| --- | --- | --- |
| `session:<id>:control` | Subscribe and send all session commands. | Interactive client. |
| `session:<id>:view` | Subscribe only. | Read-only client. |
| `session:<id>:adapter` | Connect as the adapter for one session. | Adapter process. |
| `group:<id>:control` | Control all sessions in a group. | Multi-session client. |
| `api:*` | Programmatic full access. | Administrative integration. |

Rules:

- Adapter tokens and client tokens are not interchangeable.
- Adapter tokens are bound to one session.
- The protocol only knows sessions and optional groups. Deployments map their
  own host and account concepts externally.

## Hub Embedding Interfaces

Go implementations should expose interfaces equivalent to:

```go
type PendingEvent struct {
    Type    string
    Time    time.Time
    Payload json.RawMessage
}

type Event struct {
    SessionID string
    Seq       int64
    Type      string
    Time      time.Time
    Payload   json.RawMessage
}

type EventStore interface {
    Append(ctx context.Context, sessionID string, evs []PendingEvent) (firstSeq int64, err error)
    Replay(ctx context.Context, sessionID string, afterSeq int64, fn func(Event) error) error
    LatestSeq(ctx context.Context, sessionID string) (int64, error)
}

type Authenticator interface {
    Authenticate(ctx context.Context, token string) (Principal, error)
    Authorize(ctx context.Context, p Principal, scope Scope) error
}
```

`Append` must allocate contiguous sequence numbers atomically for all events in
the batch.

## Protocol Evolution

- `protocol_version` is an integer.
- Within v1, implementations may add event types and optional fields.
- Within v1, implementations must not change existing field semantics or remove
  fields.
- Clients must ignore unknown event types.
- The `x.*` namespace is reserved for deployment-specific extensions. The Hub
  stores and forwards durable extension events without interpreting them.

## Security

- The Hub has no command execution side effects.
- Tokens should be short-lived and scoped to the minimum required capability.
- User-visible text sent by adapters should pass through secret masking.
- The Hub and EventStore must not receive known plaintext secrets from adapter
  output.
- Blob endpoints must be content-addressed and scope-checked.
- Self-hosted default listeners should bind to localhost unless explicitly
  configured otherwise.
- v1 assumes a trusted Hub. Deployments requiring an untrusted Hub need a future
  end-to-end encryption extension.

## Minimum v1 Implementation

Required for v1:

- Frames: `hello`, `hello.ack`, `event`, `command`, `command.ack`, `ping`,
  `pong`, and `error`.
- Durable events: `session.state`, `session.message`, `session.tool_call`,
  `permission.request`, `permission.decision`, and `session.error`.
- Ephemeral events: `agent.activity`, `log.tail`, and
  `session.idle_warning`.
- Commands: `session.send`, `permission.respond`, `session.interrupt`, and
  `session.stop`.
- Replay, command idempotency, adapter heartbeat, and adapter recovery states.
- EventStore and Authenticator interfaces.
- A local EventStore implementation and one adapter path.

Deferred from the minimum:

- `file.changed`.
- `presence`.
- Blob endpoint.
- Additional provider-specific fallback adapters.
- Alternate binary frame encoding.
- End-to-end encryption extension.
